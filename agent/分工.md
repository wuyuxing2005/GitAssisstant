# 迭代三 Agent 能力优化分工

## 1. 总体目标

在迭代二 Git Issue Assistant 的基础上，完成 Agent 能力优化：支持多轮对话，能够理解同一 Issue 修复任务中的上下文；从评估平台指标和 Bad Case 出发，持续优化 Agent 的推理链路、上下文管理、知识增强和工具链。


```text
Issue 修复任务
  -> Agent 多轮规划、执行、反思、验证
  -> 标准化记录工具调用、代码 diff、测试结果和 Agent Trace
  -> 评测平台计算指标并沉淀 Bad Case
  -> 分析失败原因
  -> 优化 Agent 链路、上下文、RAG 或工具环境
  -> 使用同一批样例重新评测并对比效果
```

### 1.1 现有基础

当前项目不是从零开始实现 Trace 能力，迭代二已经具备以下基础：

- Git Issue Assistant 的 `AgentState` 中已有 `trajectory`，可以记录 plan、AI 思考、工具调用请求、reflection 和 verification。
- 迭代二 Web 后端已有执行时间线 `timeline`，能展示 planner、react、tools、reflect 等节点输出。
- 评测平台已有 `AgentTrace`、`TraceEvent`、`ToolCallInfo` 数据模型，以及 Trace 上传、查询和前端 TraceViewer。

迭代三的重点不是“重新做 Trace 采集”，而是：

- 将现有 `trajectory/timeline` 统一转换为标准 `AgentTrace`。
- 补全工具事件字段，例如执行状态、耗时、错误信息、影响文件、退出码、sandbox_id。
- 打通 Git Issue Assistant 与评测平台，使 Trace 能用于 Bad Case 归因和优化前后对比。

## 2. 第一组：Agent 核心链路与能力优化组

对应原来的“Agent 推理链路组”。

### 2.1 组内定位

负责 Agent 如何理解任务、如何组织多轮上下文、如何选择 Skill 和工具、如何根据 Bad Case 改进推理策略。第一组重点解决“Agent 为什么这样做”和“失败后如何变得更好”。

### 2.2 主要职责

1. 多轮对话与上下文管理

   - 为每个 Issue 修复任务维护 `conversation_id` 或 `thread_id`。
   - 保存用户消息、Agent 回复、当前计划、已读文件、工具调用、待确认 diff、测试结果和失败原因。
   - 支持用户在同一任务中继续输入：
     - “不要修改前端，只修后端。”
     - “根据刚才的测试失败继续修。”
     - “解释为什么要改这个文件。”
   - 设计上下文压缩策略，避免长对话导致关键信息丢失。

2. Agent 架构优化

   在迭代二的 Planning-ReAct-Reflection-Verify 基础上，扩展为更可控的软件维护工作流：

   ```text
   Issue 分诊
     -> Skill 选择
     -> 修复计划生成
     -> 代码定位
     -> 修改方案生成
     -> 用户确认 diff
     -> 工具执行修改
     -> 测试验证
     -> Patch Review
     -> 成功总结或失败归因
   ```

   重点实现：

   - Issue 类型识别：bug、测试失败、CI 失败、文档问题、依赖问题等。
   - Skill Router：根据任务类型选择合适的 Skill。
   - 失败后反思：根据工具输出和测试错误调整下一步策略。
   - 终止条件优化：只有在存在代码修改、测试结果或明确验证证据时才能判定成功。

3. 工具软禁用

   - 根据用户启用/禁用的工具列表，调整 Agent 可见的工具描述。
   - Agent 规划时不主动使用被禁用工具。
   - 如果任务需要禁用工具，Agent 应说明限制，例如“当前 run_pytest 被禁用，无法执行测试验证”。
   - 软禁用只影响 Agent 规划和提示，不作为安全边界；最终拦截由第二组工具层实现。

4. Skill 机制设计

   - 定义 Skill 的文件格式、元数据和加载方式。
   - 定义 Agent 如何根据 Issue 类型、仓库信息和用户选择加载 Skill。
   - 推荐优先实现以下 Skill：
     - `bug-fix-skill`：普通 Issue 修复。
     - `test-failure-fix-skill`：测试失败修复。
     - `ci-diagnosis-skill`：CI/CD 日志诊断。
     - `patch-review-skill`：修改后自审。
     - `repo-memory-skill`：仓库级经验复用。

5. 修改前用户确认

   - Agent 在真正修改仓库文件前，应先生成修改方案或待应用 patch。
   - 修改内容需要包含：
     - 修改文件。
     - 修改原因。
     - 关键 diff。
     - 可能影响范围。
   - 用户确认后，再由工具层执行实际写入。

6. Bad Case 分析与优化

   - 从评测平台结果中找出失败样例。
   - 将失败原因归类：
     - Agent 推理问题：计划错误、文件定位错误、过早判定成功、重复无效操作。
     - 上下文问题：忘记用户约束、丢失测试错误、忽略历史修改。
     - 知识不足：不了解仓库结构、测试命令、框架约定。
     - 工具或环境问题：工具参数错误、沙箱缺依赖、测试命令不可用。
   - 如果是 Agent 方案问题，由第一组修改 prompt、状态机、Skill、反思策略或终止条件。
   - 如果是工具或环境问题，输出诊断信息交给第二组处理。

7. 现有 Trace 标准化与汇总

   - 在迭代二已有 `trajectory` 和执行时间线基础上，定义统一的 Agent Trace 标准格式。
   - 将 Planning、ReAct、Tool Call、Reflection、Verify、Patch Review、用户确认等阶段映射为统一事件。
   - 补充 Agent 主链路中的状态变化、决策原因、使用的 Skill、可见工具列表、用户补充约束和最终判定依据。
   - 汇总第二组补全后的工具调用事件，形成完整的任务执行轨迹。
   - 将完整 Agent Trace 提供给第三组，用于执行过程展示、评测指标计算和 Bad Case 数据集沉淀。

8. 自动生成 PR 描述 / 修复报告

   Agent 成功完成修复后，自动生成报告，包含：

   - Issue 摘要。
   - 根因分析。
   - 修改文件列表。
   - 关键修改说明。
   - 测试命令和测试结果。
   - 剩余风险。
   - 建议的 PR 标题和 PR 描述。
   - 该报告允许用户下载   

### 2.3 交付物

- 多轮对话上下文设计。
- Agent 架构优化说明。
- Skill 机制设计与至少 2-3 个可用 Skill。
- Bad Case 归因规则。
- 现有 `trajectory/timeline` 到标准 Agent Trace 的转换逻辑。
- 修改前确认流程设计。
- PR 描述 / 修复报告生成模块。

## 3. 第二组：工具链与执行环境组

对应原来的“工具实现组”。

### 3.1 组内定位

负责 Agent 可以调用什么工具、工具如何被安全执行、代码修改和测试如何在隔离环境中完成。第二组重点解决“Agent 能做什么”和“Agent 即使犯错也不能越权”。

### 3.2 主要职责

1. 工具硬禁用与 ToolRegistry

   建立统一工具注册表，为每个工具维护：

   - `name`：工具名称。
   - `description`：工具说明。
   - `category`：读取、搜索、修改、测试、Git、Issue、Shell 等分类。
   - `risk_level`：低、中、高。
   - `enabled_by_default`：是否默认启用。
   - `requires_confirmation`：是否需要用户确认。
   - `args_schema`：参数约束。

   工具执行前必须检查：

   - 当前任务是否启用该工具。
   - 工具参数是否合法。
   - 路径是否位于当前沙箱仓库内。
   - 是否触发危险命令规则。
   - 是否需要用户确认。

   硬禁用是最终安全边界，即使 Agent 幻觉调用了禁用工具，也必须拒绝执行。

2. Docker 沙箱

   - 为每个 Issue 修复任务创建独立工作目录，例如 `workspaces/{task_id}/repo`。
   - 在沙箱中 clone 仓库、安装依赖、运行测试和生成 diff。
   - Agent 的文件读写、命令执行、测试执行都限制在沙箱内。
   - 支持 CPU、内存、超时和网络访问限制。
   - 默认不挂载宿主机敏感文件，例如 `.env`、SSH key、API key。

3. 沙箱环境配置文件

   可以要求仓库提供指定名称的环境配置文件，例如 `.agent-sandbox.yml`。如果没有该文件，系统提示用户补充配置。

   推荐格式：

   ```yaml
   image: python:3.11
   working_dir: .

   install:
     - pip install -r requirements.txt

   test:
     - pytest

   lint:
     - ruff check .

   timeout_seconds: 300
   allow_network: true
   ```

   配置字段要求：

   | 字段 | 是否必需 | 说明 |
   | --- | --- | --- |
   | `image` | 必需 | Docker 基础镜像。 |
   | `working_dir` | 可选 | 命令执行目录，默认仓库根目录。 |
   | `install` | 可选 | 依赖安装命令列表。 |
   | `test` | 必需 | 测试命令列表。 |
   | `lint` | 可选 | 代码检查命令列表。 |
   | `timeout_seconds` | 可选 | 单次命令超时时间。 |
   | `allow_network` | 可选 | 是否允许容器联网。 |

4. 修改执行与 diff 生成

   - 支持“待确认 patch”和“确认后应用 patch”两个阶段。
   - 在实际写文件前，向 Web 层提供 diff 预览。
   - 用户确认后再调用写入工具。
   - 每次修改后生成 `git diff`，供第一组做 Patch Review，供第三组展示。

5. GitHub/GitLab Issue 接入

   - 拉取 Issue 标题、正文、评论、标签、仓库地址、目标分支。
   - 支持在 Agent 执行过程中更新 Issue 状态或添加评论。
   - 支持在任务完成后生成 PR 描述或提交说明。
   - 高风险操作，例如 commit、push、创建 PR，需要用户确认。

6. 工具调用事件字段补全与上报

   第二组不负责完整 Agent Trace 的设计和归因，但需要在迭代二已有工具输出基础上补全原始工具事件字段。每次工具调用记录：

   - `tool_name`
   - `arguments`
   - `status`
   - `result_preview`
   - `error_message`
   - `latency_ms`
   - `timestamp`
   - `sandbox_id`
   - `affected_files`
   - `exit_code`

   这些工具事件需要提供给第一组，由第一组汇总进标准 Agent Trace；同时第三组可以直接消费这些事件，用于工具调用记录展示和过程指标计算。

### 3.3 交付物

- ToolRegistry 与工具硬禁用机制。
- Docker 沙箱执行环境。
- `.agent-sandbox.yml` 配置规范。
- GitHub/GitLab Issue 接入能力。
- 工具调用事件字段补全与输出。
- 待确认 patch / 应用 patch 工具接口。

## 4. 第三组：Web 产品平台、评测平台与工程保障组

对应原来的“Web + CI/CD 组”。

### 4.1 组内定位

负责用户如何使用 Agent、如何观察 Agent 执行过程、如何管理 Bad Case 和评测结果。第三组重点解决“产品是否可用”和“优化效果如何被证明”。

### 4.2 主要职责

1. Web 页面与交互优化

   - 优化 Web 页面和人机交互操作。
   - 展示 Issue 任务列表。
   - 展示任务状态：草稿、运行中、等待用户确认、测试中、成功、失败。
   - 展示 Agent 当前计划、执行阶段和最终总结。

2. 多轮对话管理界面

   - 展示同一 Issue 下的多轮对话历史。
   - 支持用户继续补充约束或追问。
   - 支持展示当前上下文摘要。
   - 支持用户选择是否启用跨对话知识共享。

3. 自定义 LLM API

   - 支持用户配置 LLM API Key、Base URL、模型名称。
   - 支持区分 Agent 执行模型和 Judge 评测模型。
   - API Key 前端输入后应避免明文展示，后端存储需要注意安全。

4. Agent 执行过程展示

   - 在迭代二已有执行时间线和 TraceViewer 基础上，展示 Planning、Tool Call、Reflection、Verify、Patch Review 等阶段。
   - 展示工具调用记录，包括参数、结果摘要、耗时、成功/失败状态。
   - 展示 Docker 沙箱日志、测试日志和失败原因。
   - 对工具或环境问题显示参考解决方案。

5. 代码 diff 展示与用户确认

   - 展示 Agent 拟修改的文件和 diff。
   - 支持用户确认、拒绝或要求重新生成修改方案。
   - 用户确认后通知第二组工具层应用 patch。

6. Skill 添加和管理界面

   - 展示已有 Skill 列表。
   - 支持启用/禁用 Skill。
   - 支持查看 Skill 描述、适用场景和工作流程。
   - 支持上传或编辑 Skill 文件。

7. Bad Case 数据集管理

   - 保存失败 Issue、标准 Agent Trace、工具调用、diff、测试输出和失败归因。
   - 支持给 Bad Case 打标签，例如：
     - 文件定位错误。
     - 测试失败未恢复。
     - 工具参数错误。
     - 沙箱缺依赖。
     - 过早判定成功。
   - 支持重新运行 Bad Case，用于验证优化效果。

8. 评测对比展示

   - 展示优化前后任务成功率、测试通过率、工具调用准确率、平均耗时、重试恢复率等指标。
   - 支持同一批 Bad Case 在不同 Agent 版本上的结果对比。
   - 支持导出优化报告。

### 4.3 交付物

- 可用的 Web 产品。
- 多轮对话管理页面。
- Agent 执行过程可视化。
- 工具调用记录和代码 diff 展示。
- Skill 管理界面。
- Bad Case 数据集管理。
- 评测对比页面和优化报告导出。



## 6. 推荐开发顺序

1. 先梳理迭代二已有 `trajectory`、`timeline`、`AgentTrace`、Trace 上传接口和 TraceViewer。
2. 确定三组接口：任务输入输出、标准 Agent Trace、工具调用事件、Bad Case 记录、沙箱配置。
3. 第一组实现多轮上下文、Agent 新链路和 Trace 标准化转换。
4. 第二组实现 ToolRegistry、硬禁用、Docker 沙箱和工具事件字段补全。
5. 第三组实现对话、执行过程、diff 确认和标准 Trace 展示。
6. 三组共同选取 10-20 个 Bad Case，跑出优化前指标。
7. 根据 Bad Case 改进 Agent、工具和环境。
8. 重新评测并生成优化前后对比报告。
