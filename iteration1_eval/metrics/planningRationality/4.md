# Agent规划合理性评估报告

## 1. 指标选取原因

本报告旨在评估Agent在执行多步骤任务时，其行动规划的合理性。通过构建多维度的、可解释的评估体系，对系统组件的性能进行细粒度度量。

**规划合理性**被选定为核心评测指标，原因如下：

1.  **核心能力诊断**：任务规划与分解是智能Agent区别于简单指令执行器的核心能力。一个合理的规划轨迹能够反映Agent对任务的理解深度、逻辑推理能力以及工具使用的熟练度。
2.  **问题可归因**：通过分析规划轨迹，可以系统性地识别和归因Agent的典型失败模式，例如：**步骤缺失、逻辑顺序错乱、冗余操作、无效循环（死循环）以及步骤间的逻辑断层**。这些问题的识别是优化Agent策略的基础。
3.  **评估可解释性**：与最终答案的正确性相比，规划过程（由一系列思维链和工具调用组成）本身具备更强的可解释性。评估规划合理性允许我们深入到Agent的决策过程，而非仅仅评判其输出结果。
4.  **符合RAGAS理念**：该指标借鉴了RAGAS将复杂系统输出分解为多个可测量维度的思想，将“规划”这一抽象概念具体化为可计算和可评估的子维度。

## 2. 指标实现过程

`PlanningRationalityMetric` 类实现了一个**混合评估策略**，融合了基于规则的确定性计算与基于LLM的定性评判，旨在平衡评估的客观性与对复杂逻辑的涵盖能力。

### 2.1 核心评估维度
该指标从以下四个子维度综合评判规划质量，最终加权合成一个介于0到1之间的总分：
- **逻辑正确性**：评估实际步骤序列与期望或最优步骤序列在顺序和结构上的一致性。
- **数据流一致性**：评估步骤间输入输出依赖关系的正确性与连贯性，即前一步的观察结果是否被正确地用于后一步的决策。
- **效率**：评估规划中是否存在冗余步骤、不必要的循环或可合并的操作。
- **完整性**：评估规划是否涵盖了完成用户请求所必需的所有关键步骤。

### 2.2 混合评估方法

#### **2.2.1 基于规则的评估**
此部分通过确定性算法提供客观、可复现的分数。
- **规则正确性 (`_rule_correctness`)**：
  使用**最长公共子序列（LCS）** 算法，对比Agent实际调用的工具序列与“期望步骤”中定义的工具序列。计算LCS长度与期望序列长度的比值，以此量化步骤顺序的匹配度。公式为：`LCS(actual_tools, expected_tools) / len(expected_tools)`。
- **规则效率 (`_rule_efficiency`)**：
  通过三个子项综合计算：
  1.  **唯一性比率**：`len(set(actual_tools)) / len(actual_tools)`。值越低表明重复调用越多。
  2.  **循环惩罚**：通过滑动窗口检测工具调用序列中是否存在短周期重复模式（如 `[A, B, A, B]`）。若检测到循环，则施加惩罚系数（如0.7）。
  3.  **长度效率**：`min(len(actual), len(expected)) / max(len(actual), len(expected))`。惩罚步骤过多或过少。
  最终效率得分为这三项的乘积。

#### **2.2.2 基于LLM的评估**
此部分利用LLM的语义理解能力，评估规则难以覆盖的深层逻辑。
- **提示词工程 (`_build_prompt`)**：
  构建了一个结构化的提示词，包含：
  1.  **明确的评估角色与原则**。
  2.  **详细的四维度评分标准（Rubric）**，为每个维度提供了从0到1的分数档次描述。
  3.  **任务数据**：以JSON格式清晰呈现“期望计划”与“实际轨迹”。
  4.  **严格的JSON输出格式要求**，确保输出可解析。
- **解析与评分**：
  LLM根据提示词对四个维度分别打分，并提供简要的推理说明。代码通过正则表达式可靠地提取JSON结果。

#### **2.2.3 最终分数合成**
最终的规划合理性分数是规则分与LLM分的加权和，权重设计旨在强调步骤顺序（规则正确性）和深层逻辑（LLM评判）的重要性。
```python
final_score = (
    0.35 * rule_correctness +        # 强调步骤顺序匹配
    0.15 * rule_efficiency +         # 强调客观效率
    0.2 * llm_logical_correctness +  # 强调LLM判定的逻辑质量
    0.1 * llm_dataflow_consistency + # 强调数据依赖
    0.1 * llm_efficiency +           # 补充效率评估
    0.1 * llm_completeness           # 补充完整性评估
)
```

## 3. 评估结果分析

基于提供的27个测试样本（`eval_dataset.json`）及对应的评估结果（`eval_results.csv`），我们对“规划合理性”指标的表现进行深入分析。

### 3.1 整体表现概览
在全部27个任务中，规划合理性指标的**平均得分为0.785**，中位数约为0.83。分数分布呈现两极分化趋势，表明该Agent的规划能力在不同任务类型上表现不稳定。
- **高分集群（得分 ≥ 0.9）**：共9个任务，主要为简单查询（如查状态、支付单一订单）或规划路径清晰、执行准确的任务。
- **低分集群（得分 ≤ 0.7）**：共9个任务，均暴露出一项或多项明确的规划缺陷。

### 3.2 高分案例解析
以下案例展示了当Agent规划合理时，指标能够给出高分反馈：
- **任务2（得分: 1.0）**：查询订单状态。用户指令明确指向单一工具`check_order_status`，Agent直接调用，步骤与预期完全一致，无冗余。
- **任务5（得分: 1.0）**：支付指定订单。指令直接对应`pay_order`工具，参数正确，规划简洁完美。


### 3.3 低分案例归因（指标的诊断能力展示）
该指标成功识别并量化了多种规划缺陷类型，如下表所示：

| 问题类型 | 典型案例 (任务ID) | 规划合理性得分 | 失分原因与指标映射分析 |
| :--- | :--- | :--- | :--- |
| **步骤缺失** | 1, 6, 7, 9 | 0.495 - 0.795 | **最普遍缺陷**。用户明确要求“用微信支付”，但Agent在`place_order`后结束，遗漏`pay_order`。这被**完整性**和**逻辑正确性**维度严重扣分。 |
| **工具使用与参数错误** | 17, 20, 24 | 0.526 - 0.97 | Agent调用不存在工具（`get_menu`, `submit_payment`）或使用错误参数名（`product_name`, `address`, `wechat`）。导致错误观察并引发冗余纠错步骤，严重拉低**效率**和**数据流一致性**分数。 |
| **搜索策略不优** | 6, 10 | 0.495 - 0.68 | 当用户提供精确店名时，Agent未使用`restaurant_name`参数进行高效精确搜索，而是采用宽泛的`cuisine_type`搜索。步骤虽可行但不直接、不鲁棒，**效率**维度被扣分。 |
| **逻辑/数据流断层** | 11 | 0.5 | 用户查询订单`ORD-77102`，Agent却查询`ORD-77120`。其“思考”与“行动”出现矛盾，未忠实遵循用户输入，造成**数据流一致性**完全断裂和**逻辑正确性**严重失分。 |
| **冗余操作** | 7, 18 | 0.551 - 1.0 | 任务7中，Agent先用菜系搜索店铺，再用店名搜索同一店铺，属于不必要重复。这被**效率**维度捕获并扣分。 |
| **信息处理不精确** | 8 | 0.825 | 用户指定地址“仙林校区2栋A座”，Agent下单时简化为“2栋”。虽可能成功，但未严格遵循用户指令，在细节上体现了规划的不严谨性，被**逻辑正确性**和**数据流一致性**维度轻微扣分。 |

### 3.4 指标有效性交叉验证
通过对比`planning_rationality`与`task_completion`（任务完成度）指标，可以验证本指标的有效性：
- **强正相关案例**：任务1、6、9等因缺失支付步骤，两项得分均较低。这表明规划不合理直接导致了任务失败。
- **高规划分、低完成分案例（任务17）**：规划合理性得分为0.97，但任务完成度为0。分析发现，Agent规划了“获取菜单”这一合乎逻辑的步骤，但因尝试调用不存在的`get_menu`工具而失败。**这揭示了本指标的一个重要特性：它评估的是“规划过程本身的内在逻辑合理性”，而非“规划在特定工具集下的可执行性”**。这种区分有助于定位问题是出在规划逻辑上，还是出在对可用工具的认知上。

### 3.5 评估结论
1.  **指标设计成功**：`PlanningRationalityMetric` 定义的混合评估方法能够有效、多维度地度量Agent的规划质量，并准确诊断出步骤缺失、逻辑断层、冗余、参数误用等核心问题。
2.  **指标价值**：该指标不仅给出了一个总体评分，更通过其背后的多维度和混合评估方法，为理解和提升Agent的决策过程提供了可解释、可操作的诊断依据。
