# TaskCompletion

## 指标选取原因
选取任务完成度主要是以用户体验为中心，从用户的需求是否真正满足出发，使得能够包容Agent实现需求的不同路径，
更符合AI agent 行为的随机性，并且可以拆解为多步骤任务。
并且可解释性强，可以说明哪些需求完成了，哪些没完成，为什么扣分


## 实现过程
采用TaskCompletion =满足的需求数*权重 / 总需求数

参考Ragas的拆解ground_truth作原子事实用于计算准确度的方法

通过 LLM 将 user_query + ground_truth + 用户目标和期望步骤拆解得出原子需求与该需求的重要性作为权重，再通过LLM判断每个需求是否被完成

然后计算任务完成度

具体prompt：

## 评估结果分析
![alt text](image.png)
可见大部分扣分因素是由于用户要求付款而Agent缺少付款步骤步骤导致的，评分在0.7-0.8，是合理的；
值得注意的是样本8中Agent简写地址导致与expected_steps不一致，也使得评分为0.8，的确下单不一定满足用户的需求。
样本11查询了错误订单号，即使工具正确，也应该为0分，评估是合理的。样本17，支付方式wechat（没使用“微信”）导致了未知支付方式，偏离目标了，评低分是合理的。
对于样本13这样的查询不精确（拉面 ->豚骨拉面）进行了轻微扣分。

由此可见Agent整体能完成大部分需求，
但存在的问题主要是容易出现支付遗漏的问题，工具调用鲁棒性不足，address和delivery_address，wechat和微信等参数存在混淆，还可能查询错误订单号。

## 使用
通过run_eval设置Evaluator的metrics参数中添加"TaskCompletion"使用，输出为任务完成度
