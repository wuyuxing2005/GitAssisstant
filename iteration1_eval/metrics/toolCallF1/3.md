# Tool Call F1 (工具调用F1分数)

ToolCallF1 是一个用于评估 AI Agent 调用外部工具准确性的指标。通过对比 Agent 实际执行的工具调用轨迹（Actual Steps）与期望的工具调用轨迹（Expected Steps），计算两者的集合匹配情况，最终输出基于精确率（Precision）和召回率（Recall）的 F1 分数。

## 1. 评分逻辑 (Scoring Logic)

本指标使用项目内置的 LLM（`SimpleLLM`）作为裁判来进行语义级别的相似度匹配。
匹配规则如下：
1. `tool_call` 名称必须具有一一对应关系或完全匹配。
2. `input`（即工具传入参数）在语义上是等价且能完成相同目标的。即使某些语法不同（如缺少可选参数，字典键序打乱），只要实现效果等价即认为匹配成功。

**评价指标计算方式**：
- **True Positives (TP)**: 期望轨迹与实际轨迹成功匹配的工具调用数量。
- **False Positives (FP)**: 实际调用中多余（未在期望中匹配上）的工具调用数量。
- **False Negatives (FN)**: 期望调用中缺失（未在实际调用中体现）的工具调用数量。

- **Precision (精确率)** = TP / (TP + FP)
- **Recall (召回率)** = TP / (TP + FN)
- **F1 Score** = 2 * (Precision * Recall) / (Precision + Recall)

最终返回值为 `0.0` 到 `1.0` 之间的浮点数，以及包含 TP、FP、FN 及匹配细节的字典对象。

## 2. 指标选取原因 (Why ToolCallF1)

在以工具调用为核心能力的 Agent 任务中，仅用「是否完成任务」很难细粒度地刻画模型对工具的真实掌握程度。ToolCallF1 从工具调用轨迹的角度出发，兼顾「遗漏调用」与「多余调用」，具有以下优点：

1. **同时约束完整性与冗余**：通过 Precision 与 Recall 共同决定 F1，既惩罚缺少关键工具调用（FN），也惩罚无必要或错误的调用（FP），更贴近真实的「调用质量」。
2. **语义级等价判定**：借助项目内置的 `SimpleLLM` 做语义判断，即便参数顺序不同、存在可选参数缺失或轻微格式差异，只要能完成相同目标即可被视为匹配，避免了过于严格的字符串匹配导致的误判。
3. **与整体任务成功度解耦**：ToolCallF1 仅关注「工具是否被恰当地调用」，可与 `task_completion` 等结果型指标互补，帮助分析「工具调用正确但最终回答仍失败」或「工具调用不理想但任务侥幸完成」等情况。

## 3. 指标实现过程 (Implementation Details)

ToolCallF1 的核心实现位于 `iteration1_eval/metrics/toolCallF1/metric.py`，整体流程如下：

1. **样本解析**：
  - 从 `sample` 中读取 `expected_steps`（期望工具调用序列）与 `steps`（实际工具调用序列）。
  - 每个步骤包含 `tool_call`（工具名）与 `input`（参数字符串），并在内部统一封装为带有 `E0/E1...` 与 `A0/A1...` 的 ID 标记，便于后续匹配。
2. **LLM 语义匹配**：
  - 构造包含「匹配规则说明 + 期望调用列表 + 实际调用列表」的英文提示词，将其交给 `SimpleLLM`。
  - 要求 LLM 只返回一个严格的 JSON，对应字段为：`matches`（匹配到的期望/实际 ID 对）、`unmatched_expected`（未被匹配的期望调用 ID）、`unmatched_actual`（未被匹配的实际调用 ID）。
  - 通过 `extract_json_from_text` / `_parse_json_result` 对返回结果做健壮解析与清洗，容忍 LLM 包裹代码块、前后附加文本等情况。
3. **TP/FP/FN 统计与 F1 计算**：
  - 对 `matches` 中的 `(expected_id, actual_id)` 去重，防止 LLM 出现「重复配对」等幻觉。
  - 记有效匹配数为 TP，并保证其不超过期望调用数与实际调用数。
  - 通过 `FP = 实际调用数 - TP`，`FN = 期望调用数 - TP` 计算假阳性与假阴性。
  - 根据标准公式计算 Precision、Recall 与 F1，并在所有边界情况下做了特殊处理：
    - 期望与实际均为空 → 直接返回 1.0，表示在「无工具调用」场景下完全符合预期；
    - 仅有期望或仅有实际存在 → 返回 0.0，表示存在严重的遗漏或多余调用。
4. **输出结果格式**：
  - 指标最终返回一个字典，包含：`metric` 名称、`score`（F1 分数）、`precision`、`recall`、`true_positives`、`false_positives`、`false_negatives` 以及 `reasoning`（完整的匹配详情，用于后续分析与调试）。

## 4. 评估结果分析 (Evaluation Analysis)

基于 `iteration1_eval/output/eval_results.csv` 中的 27 条样本数据，可以对 ToolCallF1 的实际表现做如下观察与分析：

1. **整体工具调用质量较高**：
  - `tool_call_f1` 在绝大多数样本上接近或等于 1.0，仅有少数样本低于 0.7（例如 index 19、20、24、25）。
  - 这说明在当前评测集下，Agent 在「是否调用正确工具」和「是否漏掉关键工具调用」两个维度上整体表现较好。
2. **低分样本的典型问题模式**：
  - F1 较低（如 0.3333 或 0.6667）的样本，通常对应两类问题：
    - 期望中包含多步工具调用，但实际只完成了其中一部分，导致 FN 较多；
    - 实际调用了额外的、不必要或错误的工具，导致 FP 偏高。

## 5. 使用方法 (Usage)

在 `run_eval.py` 评估脚本中通过 `build_metric()` 注册本指标后，即可在评估配置中引用 `tool_call_f1`，与 `topic_adherence`、`task_completion` 等其他指标一同运行和汇总。
