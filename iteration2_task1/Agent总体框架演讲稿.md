# Agent 总体框架演讲稿

这一页展示的是我们 Agent 的总体框架。

整个流程从最上方的“仓库 + Issue 输入”开始。也就是说，用户需要给 Agent 一个目标代码仓库，以及一个待解决的问题描述。这个 Issue 可以是用户手动输入的文本，也可以是 GitHub Issue 编号或链接。

接下来进入 `SessionManager` 会话管理模块。这个模块主要负责准备运行环境。比如，如果输入的是远程 Git 仓库，它会先把仓库克隆到本地；如果输入的是 GitHub Issue，它会解析 Issue 内容；同时它还会为当前任务创建对应的会话线程，保证后续执行都围绕同一个仓库和同一个问题展开。

然后是 `AgentState` 初始状态。这里可以理解为 Agent 的“工作记忆”。它会保存当前仓库路径、Issue 描述、任务状态、迭代次数、修复计划、消息历史、工具调用轨迹和反思内容。后面每一次规划、推理、工具执行和验证，都会不断更新这个状态。

再往下是 `AgentOrchestrator / LangGraph`。这是整个框架的主控部分。我们使用 LangGraph 来构建状态图，也就是把 Agent 的行为拆成多个节点，并规定节点之间如何流转。这样 Agent 的执行过程不是简单的一问一答，而是一个可控制、可追踪的工作流。

进入具体执行阶段后，首先是 `planner` 节点。它会根据 Issue 生成初始修复计划，比如先阅读相关文件、搜索关键函数、定位问题、修改代码、运行测试等。这个节点的作用是让 Agent 在正式动手之前先有一个基本方向。

然后进入最核心的 `react` 节点。`react` 是整个流程的决策中心。它会根据当前状态、计划、历史消息和工具返回结果，判断下一步应该做什么。

从图中可以看到，`react` 后面有几条分支。

第一条是“需要操作仓库”。如果 Agent 判断需要进一步读取代码、搜索代码、修改文件或运行命令，就会进入 `ToolNode 工具执行`。工具执行节点会调用具体工具，比如 `read_file`、`search_code`、`patch_file`、`run_pytest` 等，对目标代码仓库进行真实操作。工具执行完成后，结果会返回给 `react`，让 Agent 根据新的证据继续推理。

第二条是“已有修改或声明成功”。如果 Agent 已经完成了代码修改，或者模型输出了成功信号，就不会直接结束，而是进入 `verify 自动验证` 节点。这个节点会检查是否真的有成功证据，比如是否运行测试通过，是否发生了有效代码修改，Issue 中要求的文件是否已经存在等。只有验证通过，才会进入 `SUCCESS` 状态。

第三条是“未通过”。如果验证节点发现证据不足，或者测试没有通过，就会回到 `react`。也就是说，Agent 会继续分析失败原因，并尝试进一步修复。

第四条是“无进展或重复输出”。如果 Agent 连续几轮没有调用工具，或者输出内容重复，就会进入 `reflect 失败反思` 节点。这个节点会总结当前为什么没有推进，比如缺少哪些证据、下一步应该调用什么工具。反思完成后，再回到 `react` 继续执行。

最后一条是“失败或超出轮数”。如果 Agent 明确判断任务无法继续，或者迭代次数超过最大限制，就会进入 `FAILED` 状态，任务结束。

所以总结来看，这张图体现了我们 Agent 框架的核心思想：它不是让大模型一次性给出答案，而是用 LangGraph 构建了一个带状态的自动修复闭环。`AgentState` 负责保存上下文，`react` 负责决策，`ToolNode` 负责真实执行，`verify` 负责验证结果，`reflect` 负责失败后的自我修正。

最终，Agent 会在这些节点之间循环流转，直到修复成功、修复失败，或者达到最大迭代次数。
