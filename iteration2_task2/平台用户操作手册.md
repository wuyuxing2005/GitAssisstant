# AgentEval 平台用户操作手册

本文档面向平台使用者，重点说明“在平台上怎么操作”以及“每个选项有什么用”。内容依据 `iteration2_task2` 当前项目实现整理，覆盖目前前端界面中已提供的主要功能。

## 1. 平台入口与整体导航

进入系统后，左侧边栏提供 5 个主要页面：

- `Dashboard`：查看任务总览、任务列表，并执行运行、删除、对比选择等操作。
- `Task Builder`：创建新的评测任务，配置评测方式、指标和策略。
- `Analysis`：查看单个任务的详细分析结果，包括概览、图表、Trace 和报告导出。
- `Comparison`：对多个任务的评测结果进行横向对比。
- `Datasets`：管理评测数据集，包括上传、预览和删除。

右上角会显示系统在线状态；页面右上和右下会用通知提示操作是否成功，例如任务创建成功、任务运行失败、数据集上传完成等。

---

## 2. 推荐使用顺序

如果你是第一次使用这个平台，建议按下面顺序操作：

1. 先进入 `Datasets` 页面上传或确认已有数据集。
2. 再进入 `Task Builder` 页面创建评测任务。
3. 回到 `Dashboard` 页面启动任务运行。
4. 点击任务进入 `Analysis` 页面查看结果。
5. 勾选多个已完成任务，进入 `Comparison` 页面做横向比较。
6. 在 `Analysis` 页面导出 JSON 或 Markdown 报告。

## 3. `Dashboard` 页面：任务总览与快捷操作

`Dashboard` 是平台首页，适合查看当前有哪些任务、哪些任务已完成、哪些任务正在运行。

### 3.1 顶部汇总卡片有什么用

页面顶部会显示若干汇总卡片，用于快速了解平台当前状态：

- **Total**：当前系统中的任务总数。
- **Running**：当前状态为运行中的任务数量。
- **Completed**：当前已经完成的任务数量。
- **Custom Metrics**：所有任务中累计配置过的自定义指标数量。

这些卡片适合用来快速把握当前评测工作量和活跃程度。

### 3.2 任务列表怎么看

任务表格中，每一行代表一个评测任务，包含以下信息：

- **Compare**：勾选后可把该任务加入对比分析候选。
- **Task**：任务名称与说明；如果任务配置过自定义指标，会显示一个数量徽标。
- **Agent**：该任务评测的 Agent 版本。
- **Dataset**：该任务使用的数据集。
- **Modes**：评测模式，表示本次关注结果、过程，或两者都看。
- **Strategy**：本任务采用的指标组合策略模板。
- **Status**：任务当前状态。
- **Updated**：最近更新时间。
- **Actions**：可执行的操作按钮。

### 3.3 状态的含义

平台当前支持以下任务状态：

- **Draft**：草稿，任务已创建但通常尚未正式执行。
- **Scheduled**：已排队，表示准备执行。
- **Running**：运行中，表示后端正在计算评测结果。
- **Completed**：已完成，可查看分析结果并参与对比。
- **Failed**：失败，通常表示运行时缺少数据集、Trace 或其他依赖。

### 3.4 你能做哪些操作

在任务列表中，你可以直接完成这些操作：

- **点击任务行**：选中该任务，并自动跳转到 `Analysis` 页面查看详情。
- **勾选 Compare**：把任务加入比较范围，之后可去 `Comparison` 页面查看结果。
- **点击 Run**：立即运行该任务。
- **点击 Delete**：删除该任务。

### 3.5 `Run` 按钮有什么用

- 用于请求后端立即执行该评测任务。
- 运行过程中按钮会变成加载状态，避免重复点击。
- 如果后端执行失败，页面会显示错误提示，例如缺少数据集文件、结果文件或其他运行依赖。

### 3.6 `Delete` 按钮有什么用

- 用于删除当前任务。
- 删除后该任务会从任务列表和比较列表中移除。
- 如果该任务原本是当前正在查看的任务，分析页会失去对应选择对象。

## 4. `Task Builder` 页面：创建评测任务

`Task Builder` 页面用于新建任务。这里是平台最核心的配置入口。

创建任务时，你需要依次填写名称、说明、基础配置、评测模式、评测方法、评测维度、策略模板和指标配置。

### 4.1 基础字段怎么填

#### `Task Name`

- 任务名称。
- 用于在任务列表、分析页、导出报告中区分不同评测任务。
- 建议写成便于识别的形式，例如：`v2.0-customer-support-质量评估`。

#### `Description`

- 任务说明。
- 用于记录这次评测的目的，例如“比较新旧 Agent 在客服问答上的差异”。
- 可选，但建议填写，后续回看任务时更容易理解背景。

#### `Agent Version`

- 选择要评测的 Agent 版本。
- 当前系统内置的版本选项来自后端元数据，例如：`v1.3.0`、`v1.4.0-rc1`、`v2.0.0-beta`。
- 如果你想比较不同版本效果，通常需要分别创建多个任务，每个任务选择不同版本。

#### `Dataset`

- 选择本次评测使用的数据集。
- 下拉选项来自 `Datasets` 页面中已存在的 `.jsonl` 数据集文件。
- 如果这里没有可选数据集，说明你还没有上传数据集，需先去 `Datasets` 页面补充。

#### `Initial Status`

- 任务创建时的初始状态。
- 可选值：`draft`、`scheduled`、`running`、`completed`、`failed`。
- 对普通用户来说，最常用的是：
  - `draft`：先保存任务，稍后再运行。
  - `scheduled`：表示这是一个待执行任务。
- 虽然界面允许选择其他状态，但从使用习惯上更建议让系统在运行后自动更新状态，而不是手工把任务直接标记为 `completed` 或 `failed`。

### 4.2 `Modes`：评测模式怎么选

这里用于决定你想从哪个角度评估 Agent。

#### `Result-oriented`

- 面向结果的评估。
- 重点看最终回答好不好、准不准、是否符合预期。
- 适合只关注输入和输出的场景。

#### `Process-oriented`

- 面向过程的评估。
- 重点看 Agent 在中间过程中的推理、工具调用、执行步骤是否合理。
- 适合希望检查“Agent 是怎么得出答案的”这一类场景。

#### 如何选择

- 只想看最终回答质量：选 `Result-oriented`。
- 想分析工具调用或推理链路：选 `Process-oriented`。
- 想同时看结果和过程：两个都选。

### 4.3 `Methods`：评测方法怎么选

这里用于决定“分数是怎么得到的”。

#### `Explicit Metric`

- 显式指标。
- 适合可以明确计算的指标，例如耗时、工具准确率、任务成功率、Token 用量等。
- 特点是规则明确、结果稳定，适合做量化比较。

#### `LLM-as-a-Judge`

- 让模型担任评委，对回答或过程进行打分。
- 适合质量、幻觉、安全性、推理质量等更主观的指标。
- 如果你要配置自定义 Judge 指标，通常需要选择这个方法。

#### 如何选择

- 看效率、成功率、调用准确率：偏向选 `Explicit Metric`。
- 看质量、安全、推理表现：偏向选 `LLM-as-a-Judge`。
- 想做综合评估：两个都选。

### 4.4 `Dimensions`：评测维度怎么选

这里决定你更关注哪一类能力。

#### `Quality`

- 质量维度。
- 关注回答是否正确、完整、相关，或者工具使用是否合理。

#### `Safety`

- 安全维度。
- 关注输出是否安全、是否有幻觉、是否违反预期约束。

#### `Performance`

- 性能维度。
- 关注延迟、响应时间、Token 使用量等效率指标。

#### 如何选择

- 想衡量回答效果：选 `Quality`。
- 想重点控制风险：选 `Safety`。
- 想关注用户体验和资源消耗：选 `Performance`。
- 若想全面评估，可以三个都选。

### 4.5 `Strategy Template`：策略模板有什么用

策略模板是一组预先定义好的指标组合。选择模板后，系统会自动帮你勾选一批默认指标。

当前平台内置三种模板：

#### `Balanced Default`

- 平衡型默认策略。
- 同时兼顾质量、安全和性能。
- 默认适合大多数通用评测任务。
- 系统默认会偏向选择：
  - `Answer Correctness`
  - `Safety`
  - `Latency`

#### `Trace First`

- 过程优先策略。
- 更关注推理过程和工具使用质量。
- 适合需要检查 Agent 中间执行链路的场景。
- 常搭配：
  - `Tool Accuracy`
  - `Reasoning Quality`
  - `Token Usage`

#### `Safety Guardrail`

- 安全优先策略。
- 更适合面向用户的 Agent 场景，强调输出是否安全、是否有幻觉。
- 常搭配：
  - `Safety`
  - `Hallucination Risk`
  - `Response Time`

#### 使用建议

- 不确定怎么配时，先选 `Balanced Default`。
- 需要复盘 Agent 推理和工具行为时，选 `Trace First`。
- 要上线用户场景、担心风险时，选 `Safety Guardrail`。

### 4.6 `Built-in Metrics`：内置指标怎么选

这里可以手动勾选本次任务要计算的内置指标。每个指标卡片都会显示其所属维度和方法。

当前项目内置的主要指标包括：

#### 质量类

- **Answer Correctness**：评估回答是否正确，适合结果质量评估。
- **Faithfulness**：评估回答是否忠于给定上下文，是否“有依据”。
- **Task Success Rate**：评估任务完成率。
- **Tool Accuracy**：评估工具选择和参数使用是否正确。
- **Reasoning Quality**：评估中间推理过程是否清晰、合理。

#### 安全类

- **Hallucination Risk**：评估回答是否存在幻觉或编造内容。
- **Safety**：评估回答是否安全、是否包含不当内容。

#### 性能类

- **Latency**：评估完整请求耗时。
- **Response Time**：评估用户首次可感知的响应速度。
- **Token Usage**：评估平均 Token 消耗。

#### 使用建议

- 希望得到更少更聚焦的结果：只勾选与你目标最相关的几个指标。
- 希望全面比较不同版本：可在多个任务中保持指标组合一致，便于横向对比。
- 切换策略模板后，系统会自动更新默认勾选的内置指标；你仍然可以手动增删。

### 4.7 `Custom Metrics`：自定义指标怎么用

如果内置指标不够用，你可以添加自定义指标。

点击 `Add Metric` 后会新增一张自定义指标卡片，需要填写以下内容：

#### `metric_key`

- 自定义指标的内部标识。
- 用于系统识别，建议使用英文、小写、下划线形式，例如 `dialog_empathy`。

#### `Metric label`

- 页面展示名。
- 会显示在结果页、对比页和导出报告中。

#### `dimension`

- 指标所属维度：`quality`、`safety`、`performance`。
- 决定该指标在图表中的归类位置。

#### `method`

- 指标使用的评测方法：`explicit` 或 `judge`。
- 如果选择 `judge`，页面会额外出现提示词配置区域。

#### `Description`

- 指标说明。
- 建议写清楚你希望这个指标评价什么。

#### 删除自定义指标

- 点击指标卡片右上角的 `×` 可移除当前自定义指标。

### 4.8 `配置自定义 Judge 提示词` 有什么用

当某个自定义指标的方法选为 `judge` 时，可以勾选 `配置自定义 Judge 提示词`，进一步定制模型如何打分。

你会看到以下配置项：

#### `选择预设模板`

当前系统提供多种预设 Judge 模板：

- `默认评估`：通用质量评分模板。
- `推理质量评估`：更适合评估推理过程。
- `幻觉检测`：更适合评估是否出现幻觉。
- `安全性评估`：更适合评估内容是否安全。
- `工具使用质量`：更适合评估工具调用是否合理。
- `代码质量评估`：更适合评估生成代码质量。

#### `自定义提示词（可选，留空则使用模板）`

- 你可以直接写一段自己的 Judge 提示词，覆盖默认模板。
- 支持在提示词中使用这些变量：
  - `{user_input}`
  - `{response}`
  - `{reference}`
  - `{criteria}`
- 如果留空，系统会使用你选中的预设模板。

#### `评估标准`

- 这里会显示当前模板附带的评分标准。
- 你可以手动增加、删除标准，让 Judge 更贴合你的业务目标。

#### `添加`

- 输入“标准名称”和“标准描述”后点击 `添加`，可增加新的评估标准。
- 例如你可以添加 `tone`、`empathy`、`policy_compliance` 等自定义标准。

### 4.9 创建任务

全部配置完成后，点击页面底部的 `Create Evaluation Task`：

- 创建成功后，系统会提示成功消息。
- 页面会自动回到 `Dashboard`。
- 新任务会出现在任务列表中。

## 5. `Analysis` 页面：查看单个任务结果

`Analysis` 页面用于查看一个任务的详细分析结果。

通常你有两种进入方式：

- 在 `Dashboard` 中点击任务行自动跳转。
- 在左侧切到 `Analysis` 页面后查看当前选中的任务。

如果没有选中任务，页面会提示你先从表格中选择一个任务。

### 5.1 页面顶部的导出区域怎么用

如果该任务已经有结果，页面右上会出现导出选项：

#### 导出格式下拉框

当前前端可选：

- `JSON`：适合程序处理、保存原始结构化结果。
- `Markdown`：适合阅读、汇报、沉淀为分析文档。

#### `Export`

- 点击后会下载该任务的评测报告。
- 当前前端只提供 `JSON` 和 `Markdown` 两种常用格式。
- 后端虽然保留了 `pdf` 导出接口占位，但目前尚未实现 PDF 导出。

### 5.2 `Overview` 标签：概览页怎么看

`Overview` 是默认标签，适合快速了解任务配置和运行结果。

#### `Scorecard`

- 用卡片形式展示聚合后的关键分数。
- 适合快速看这个任务总体表现如何。
- 如果任务还没运行，会提示先运行任务。

#### `Task Configuration`

这里会回显本次任务的主要配置：

- `Agent Version`
- `Dataset`
- `Evaluation Modes`
- `Methods`
- `Dimensions`
- `Strategy`

适合在看结果时确认“这次评测到底是按什么配置跑的”。

#### `Run Summary`

- 展示本次评测的文字总结。
- 如果结果里带有时间线，还会显示流程进度组件，用于展示评测阶段的推进情况。

#### `Metric Results`

- 以表格形式逐项列出所有指标。
- 包含：
  - `Metric`：指标名称
  - `Value`：指标值
  - `Category`：所属类别/维度
- 适合精确查看每个指标的得分或数值表现。

#### `Execution Logs`

- 展示本次任务执行时截取的一部分日志。
- 适合排查运行过程问题，例如缺少数据、某个计算阶段失败等。

### 5.3 `Charts` 标签：图表怎么看

切换到 `Charts` 标签后，会看到两类图表：

#### `Dimension Radar Chart`

- 雷达图按 `quality`、`safety`、`performance` 三个维度聚合指标分数。
- 适合快速判断该任务在哪个维度更强、哪个维度更弱。

#### `Metric Breakdown`

- 柱状图逐项展示各指标值。
- 适合看某几个指标是优势项还是短板项。

### 5.4 `Traces` 标签：执行链路怎么看

切换到 `Traces` 标签后，会进入执行 Trace 查看区域。

这里适合回答下面这些问题：

- Agent 在中间做了哪些步骤？
- 有没有调用工具？
- 哪一步耗时长、哪一步可能异常？

如果对应任务没有可用 Trace 数据，页面内容会受到后端返回数据限制，可能无法展示完整链路。

## 6. `Comparison` 页面：对多个任务做横向比较

这个页面用于比较多个任务之间的差异。

### 6.1 如何开始比较

1. 先回到 `Dashboard` 页面。
2. 在任务表格左侧 `Compare` 列勾选你想比较的任务。
3. 再进入 `Comparison` 页面查看结果。

### 6.2 页面上的三种展示方式

#### `Cards`

- 适合看多个任务的概况。
- 每张卡片会展示：
  - 任务名称
  - Agent 版本 / 数据集
  - Scorecard 聚合结果
  - 各项指标分数

适合先做总览式比较。

#### `Radar`

- 用雷达图比较任务在质量、安全、性能三个维度上的表现。
- 最适合比较两个任务。
- 如果你选了超过两个任务，页面会提示雷达图只展示前两个任务。

#### `Bar Chart`

- 用柱状图比较两个任务在各具体指标上的差异。
- 同样最适合比较两个任务。
- 如果选择超过两个任务，柱状图仍只展示前两个任务。

### 6.3 使用建议

- 想对比多个历史任务的总体情况：先看 `Cards`。
- 想对比两个版本在三大维度上的差别：看 `Radar`。
- 想看具体是哪些指标拉开差距：看 `Bar Chart`。

## 7. `Datasets` 页面：上传、预览和删除数据集

这个页面用于管理评测数据集。任务创建页面里的 `Dataset` 下拉选项也来源于这里。

### 7.1 页面能看到什么

数据集列表会展示：

- `Name`：数据集名称。
- `Records`：数据条数。
- `Size`：文件大小。
- `Modified`：最近修改时间。
- `Actions`：查看或删除操作。

### 7.2 如何上传数据集

点击右上角 `Upload Dataset` 按钮后，会展开上传区域。

你可以：

- 直接把文件拖拽到上传区域。
- 或点击上传区域，打开文件选择器选择文件。

### 7.3 上传文件有哪些要求

当前页面和后端都要求：

- 文件格式必须是 `.jsonl`。
- 每一行都必须是一个 JSON 对象。
- 必填字段：`user_input`。
- 常见可选字段：
  - `response`
  - `reference`
  - `retrieved_contexts`

如果文件不满足要求，上传会失败，页面会显示错误原因。

### 7.4 `View` 按钮有什么用

点击 `View` 后会弹出数据集详情窗口，显示：

- 数据条数
- 文件大小
- 创建时间
- 修改时间
- 前若干条样本预览

适合在建任务前确认数据集内容是否正确。

### 7.5 `Delete` 按钮有什么用

- 用于删除该数据集文件。
- 删除后，这个数据集将不再出现在 `Task Builder` 的数据集下拉框中。
- 如果已有任务引用该数据集，后续运行时可能报错，因此删除前要确认是否仍被使用。

## 8. 常见使用场景示例

### 场景一：比较两个 Agent 版本谁更好

1. 进入 `Datasets`，确认已有目标数据集。
2. 进入 `Task Builder`，创建第一个任务：
   - `Agent Version` 选 `v1.3.0`
   - 数据集选同一个数据集
   - 策略选 `Balanced Default`
3. 再创建第二个任务：
   - `Agent Version` 选 `v2.0.0-beta`
   - 其他配置尽量保持一致
4. 回到 `Dashboard`，分别点击两个任务的 `Run`。
5. 运行完成后，在 `Compare` 列勾选这两个任务。
6. 进入 `Comparison` 页面，切换 `Radar` 或 `Bar Chart` 查看差异。

### 场景二：重点检查 Agent 的推理过程和工具调用

1. 创建任务时在 `Modes` 中勾选 `Process-oriented`。
2. 在 `Methods` 中保留 `Explicit Metric` 和 `LLM-as-a-Judge`。
3. 在 `Strategy Template` 中选择 `Trace First`。
4. 需要时补充一个自定义指标，例如“工具结果整合质量”。
5. 运行任务后进入 `Analysis` 的 `Traces` 和 `Charts` 页面重点查看。

### 场景三：重点做安全评估

1. 创建任务时在 `Dimensions` 中至少勾选 `Safety`。
2. 选择 `Safety Guardrail` 模板。
3. 保留 `Safety`、`Hallucination Risk` 等指标。
4. 如有需要，增加一个自定义 Judge 指标并选择“安全性评估”模板。
5. 运行后在 `Overview` 中查看安全类指标，在 `Comparison` 中横向比较不同版本的风险表现。

## 9. 当前已实现但需要注意的地方

根据当前项目实现，使用时建议注意这些点：

- 平台支持创建、运行、删除、分析、对比和数据集管理，但**前端暂未提供任务编辑功能**。
- 对比页虽然可选择多个任务，但 `Radar` 和 `Bar Chart` 最适合比较前两个任务。
- 报告导出当前以 `JSON` 和 `Markdown` 为主，**PDF 尚未实现**。
- 数据集缺失、格式不正确，或者运行依赖文件缺失时，任务可能进入失败状态。
- `Traces` 标签页是否展示完整内容，取决于后端是否有对应 Trace 数据。

## 10. 一句话总结每个页面适合做什么

- `Dashboard`：看全局、点运行、删任务、选对比。
- `Task Builder`：新建任务并配置评测方式。
- `Analysis`：看单个任务的详细结果并导出报告。
- `Comparison`：比较多个任务，尤其适合两个版本做横向评估。
- `Datasets`：上传和检查评测数据集。
